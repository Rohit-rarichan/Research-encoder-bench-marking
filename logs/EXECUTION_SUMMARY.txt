3️⃣  ADVANCED TRAINING TECHNIQUES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A. LOSS FUNCTION: Cross-Entropy with Inverse Frequency Weighting
   ───────────────────────────────────────────────────────────────
   Problem Solved: Extreme class imbalance (car 141M vs bicycle 2.6M)
   
   Mechanism:
     weight[c] = total_pixels / (num_classes * count[c])
     Normalized by class count, scaled by class frequency
   
   Effect:
     - Rare classes (bicycle, motorcycle): 2-4× higher gradient contribution
     - Common classes (car, surface): 0.01-0.08× reduced contribution
     - Prevents model from ignoring minority classes
     - Balanced training signal across all semantic categories


B. OPTIMIZER: AdamW (Adam with Decoupled Weight Decay)
   ──────────────────────────────────────────────────
   Config:
     Learning Rate: 6e-5 (0.00006)
     Beta1: 0.9 (momentum for first moment estimate)
     Beta2: 0.999 (RMSprop for second moment estimate)
     Weight Decay: 0.01 (L2 regularization applied separately)
   
   Advantages:
     - Adaptive per-parameter learning rates
     - Faster convergence than SGD
     - Better generalization than standard Adam (due to decoupled weight decay)
     - Robust to learning rate choice


C. LEARNING RATE SCHEDULE: WarmupThenSlash
   ──────────────────────────────────────────
   Stage 1 - Warmup (Epochs 1-5):
     Linear increase from 0 to 6e-5
     Prevents gradient shock from random initialization
     
     Epoch 1: LR = 1.2e-5
     Epoch 2: LR = 2.4e-5
     Epoch 3: LR = 3.6e-5
     Epoch 4: LR = 4.8e-5
     Epoch 5: LR = 6.0e-5
   
   Stage 2 - Exponential Decay (Every 10 epochs):
     Slash factor: 0.5 (multiply by 0.5)
     
     Epochs 6-15:   LR = 6.0e-5
     Epochs 16-25:  LR = 3.0e-5
     Epochs 26-35:  LR = 1.5e-5
     Epochs 36-45:  LR = 7.5e-6
     Epochs 46-55:  LR = 3.75e-6
     Epochs 56-100: Continues halving
   
   Rationale:
     - Warmup stabilizes training from random init
     - Exponential decay allows exploration then fine-tuning
     - Gradual transitions prevent oscillations


D. EARLY STOPPING: Patience=20 Epochs
   ───────────────────────────────────
   Monitor: Validation Loss
   Patience: 20 epochs without improvement
   
   Logic:
     - Every epoch: If val_loss < best_val_loss → save checkpoint
     - If val_loss >= best_val_loss → increment counter
     - If counter == 20 → stop training
   
   Previous Issue (FIXED):
     Old: patience=5 → stopped at epoch 5-7 (premature termination)
     New: patience=20 → typical convergence 20-35 epochs, buffer for noise
   
   Benefit: Training finds optimal point without manual epoch selection


E. MIXED PRECISION TRAINING (Automatic Mixed Precision / AMP)
   ──────────────────────────────────────────────────────────
   How It Works:
     - Forward pass: FP16 for fast ops (convolution, matrix mult)
     - Backward pass: FP16 for memory efficiency
     - Loss scaling: Prevent FP16 gradient underflow
   
   Performance Gains:
     Speed:     30-40% faster training
     Memory:    20-30% less VRAM usage
     Accuracy:  <0.1% loss vs full FP32 training
   
   Implementation: CUDA GradScaler with autocast context


F. PER-EPOCH VALIDATION
   ─────────────────────
   Every Epoch:
     1. Run inference on 491 validation samples
     2. Compute validation loss (CE with class weights)
     3. Calculate mIoU on 6 relevant classes
     4. Store metrics in history
     5. Save checkpoint if loss improved
     6. Check early stopping condition
     7. Print epoch summary
   
   Output: Training history with per-epoch metrics


G. CLASS WEIGHTING FOR BALANCE
   ────────────────────────────
   Inverse Frequency Formula:
     weight[c] = total_pixels / (num_classes * pixels_in_class_c)
     
   Then normalized:
     weight = weight / sum(weight) * num_classes
   
   Result:
     Rare classes get higher gradient contribution
     Common classes get downweighted
     Model learns all semantic categories


H. REPRODUCIBILITY: Deterministic Training
   ───────────────────────────────────────
   Seed: 42 (applied to all random operations)
   - torch.manual_seed(42)
   - np.random.seed(42)
   - torch.cuda.manual_seed_all(42)
   
   CUDA Settings:
   - deterministic=True (slower but exact same results)
   - benchmark=False (disable autotuning)
   
   Guarantee: Same code + same seed = identical results across runs

BACKUP AUTOMATION:

After Each Model Completes:
  bash backup_checkpoints.sh

This script:
  1. Finds all .pth files in model directories
  2. Creates timestamped backup subdirectory (e.g., model_20260424_1400)
  3. Copies all checkpoints with verification
  4. Copies results.json for record-keeping
  5. Verifies file integrity (loads checkpoints to verify)
  6. Sets backups to read-only mode (chmod 444)
  7. Displays backup summary


CHECKPOINT RECOVERY:

If Training Interrupted:
  Training auto-detects and resumes from checkpoint_best.pth
  No manual action needed - just restart the training script

If Checkpoint Lost:
  1. Check backups: ls outputs/checkpoints_backup/*/checkpoint_best.pth
  2. Restore: cp outputs/checkpoints_backup/*/checkpoint_best.pth outputs/MODEL/
  3. Resume: python train_nuimages.py --model MODEL_NAME

If Everything Lost:
  Last resort: retrain from scratch (40-50 GPU hours per model)


PROTECTION MECHANISMS:

1. Backup directory set to read-only:
   chmod 444 outputs/checkpoints_backup/*
   
2. Timestamped subdirectories prevent overwriting:
   outputs/checkpoints_backup/model_YYYYMMDD_HHMM/
   
3. Integrity verification:
   Loads each checkpoint to verify not corrupted
   
4. Script prevents accidental mass deletion:
   Use backup_checkpoints.sh for all management

ISSUE #1: Learning Rate Scheduler Per-Batch Bug
  ✓ Status: FIXED
  
  Problem: lr_scheduler.step() was inside the batch training loop
  Effect:  With 586 batches per epoch:
            - After epoch 1: current_epoch jumped from 0 to 586
            - Learning rate: 6e-5 × (0.5^58) ≈ 7e-40
            - Model completely froze, gradient updates became 0
            - mIoU stuck at 0.06 (near-random predictions)
  
  Root Cause: Misunderstanding of PyTorch LR scheduler
  
  Solution: Moved lr_scheduler.step() from inside batch loop
            to once-per-epoch AFTER validation
  
  File: train_nuimages.py
  Line: 506 (added lr_scheduler.step() here)
  

ISSUE #2: Checkpoint Loading Reset
  ✓ Status: FIXED
  
  Problem: After loading checkpoint, training restarted from epoch 0
  Effect:  - Optimizer state reset (lost momentum/RMSprop)
           - Scheduler state reset (lost epoch count)
           - 30-40 hours of prior training wasted
           - Model retrained with bad initialization
  
  Solution: Rewrote checkpoint loading to:
            1. Load full checkpoint_best.pth (not weights-only)
            2. Restore optimizer state via load_state_dict()
            3. Restore scheduler state
            4. Set start_epoch = len(history) to resume from correct epoch
  
  File: train_nuimages.py
  Lines: 410-475


ISSUE #3: Early Stopping Too Aggressive
  ✓ Status: FIXED
  
  Problem: Early stopping patience was only 5 epochs
  Effect:  - Training stopped at epoch 5-7
           - Models never converged
           - Lost 40+ hours of potential training per model
  
  Solution: Increased patience from 5 to 20 epochs
            - Typical convergence: 20-35 epochs
            - Allows natural noise in val loss
            - Prevents premature termination
  
  File: train_nuimages.py
  Line: 398


ISSUE #4: ConvNeXt-B Out of Memory (OOM)
  ✓ Status: FIXED
  
  Problem: ConvNeXt-B couldn't fit in 24GB VRAM with batch_size 6
  Error:   CUDA out of memory, tried to allocate 264 MiB
  
  Solution: Reduced batch size from 6 → 4
            - Similar to SegFormer and MiT-B0
            - Both are memory-intensive models
            - Reduces training speed but eliminates crashes
  
  File: train_all_models.sh
  Config: Conditional batch size logic
